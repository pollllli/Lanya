# 代码审查流程

## 审查目的
- 确保代码质量和一致性
- 发现并修复潜在的bug和安全问题
- 确保代码符合项目的编码规范
- 促进团队成员之间的知识共享
- 提高代码的可维护性和可扩展性

## 审查流程

### 1. 提交代码前
- 运行 `npm run lint` 检查代码风格
- 运行 `npm run typecheck` 检查TypeScript类型
- 确保所有测试通过
- 编写清晰的提交信息，描述变更的内容和原因

### 2. 提交Pull Request
- 标题清晰描述变更内容
- 正文详细说明变更的原因、实现方式和影响
- 附上相关的issue链接（如果有）
- 确保代码已经通过本地测试

### 3. 代码审查
- **代码风格**：检查是否符合ESLint和Prettier的规范
- **类型安全**：检查TypeScript类型定义是否正确
- **逻辑正确性**：检查代码逻辑是否正确，是否有潜在的bug
- **性能**：检查是否有性能问题，如不必要的渲染、重复计算等
- **安全性**：检查是否有安全问题，如SQL注入、XSS攻击等
- **可维护性**：检查代码是否易于理解和维护，是否有适当的注释
- **测试覆盖**：检查是否有足够的测试覆盖

### 4. 审查反馈
- 提出具体的问题和建议
- 提供改进的具体示例
- 保持建设性的语气
- 确保反馈清晰明确

### 5. 修复和更新
- 根据审查反馈进行修改
- 重新运行测试和lint检查
- 更新Pull Request描述，说明修复的内容

### 6. 合并代码
- 确保所有审查意见都已解决
- 确保所有测试都已通过
- 合并代码到主分支

## 审查标准

### 代码风格
- 遵循ESLint和Prettier的规则
- 使用一致的命名约定
- 保持适当的缩进和空行
- 避免过长的函数和文件

### 类型安全
- 正确使用TypeScript类型
- 避免使用 `any` 类型
- 为函数参数和返回值添加类型注解
- 为复杂数据结构定义接口

### 逻辑正确性
- 处理所有可能的边界情况
- 避免逻辑错误和死代码
- 确保错误处理机制完善
- 验证输入数据的有效性

### 性能
- 避免不必要的渲染和计算
- 合理使用React的性能优化特性（如useMemo、useCallback）
- 优化网络请求和存储操作
- 避免内存泄漏

### 安全性
- 防止SQL注入和XSS攻击
- 安全处理用户输入
- 保护敏感信息
- 遵循安全最佳实践

### 可维护性
- 编写清晰的注释和文档
- 遵循模块化和组件化原则
- 使用有意义的变量和函数名
- 避免重复代码

### 测试覆盖
- 为核心功能编写单元测试
- 测试边界情况和错误处理
- 确保测试覆盖率达到合理水平

## 工具和资源
- **ESLint**：检查代码风格
- **Prettier**：格式化代码
- **TypeScript**：类型检查
- **Jest**：单元测试
- **React Testing Library**：组件测试

## 常见问题和解决方案

### 代码风格问题
- **问题**：代码不符合ESLint规则
- **解决方案**：运行 `npx eslint --fix` 自动修复

### 类型错误
- **问题**：TypeScript类型错误
- **解决方案**：添加正确的类型注解或类型定义

### 性能问题
- **问题**：不必要的渲染或计算
- **解决方案**：使用useMemo、useCallback等React hooks

### 安全问题
- **问题**：潜在的安全漏洞
- **解决方案**：遵循安全最佳实践，使用安全的库和方法

### 可维护性问题
- **问题**：代码难以理解或维护
- **解决方案**：重构代码，添加注释，遵循模块化原则
